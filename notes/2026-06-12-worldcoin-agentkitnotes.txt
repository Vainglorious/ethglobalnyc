2026-06-12 — Worldcoin / World Agent Kit notes

WHAT AGENTKIT IS
  AgentKit (Beta) is an extension of the x402 payment protocol from World. Its job:
  let a server distinguish HUMAN-BACKED agents from bots/scripts. A real human vouches
  for an agent wallet; the server can then resolve that agent to an anonymous human
  identifier ("humanId") at request time and apply access / payment rules.
  Docs saved at: worldcoin/docs/agent-kit-integrate.md
  Source: https://docs.world.org/agents/agent-kit/integrate

THE TWO IDENTITIES (the thing that was confusing, now clear)
  1. Agent wallet (private key + address): just a normal EVM keypair. NOT issued by
     Worldcoin. You generate it yourself (we used `cast wallet new`). Each agent CAN
     have its own keypair. The key signs the agent's requests (eip191).
  2. Your World ID (World App): never stored in a .env, no API key. It is used LIVE /
     interactively at registration time — you approve in the World App on your phone.
     That approval is what binds an agent address to YOU (a real human).

CLI FACTS LEARNED
  - register defaults to --auto: submits to relay https://x402-worldchain.vercel.app
  - register --manual: prints the call data instead of submitting (inspect what's signed)
  - register has NO private-key flag: registration proves a HUMAN vouches; it does not
    use the agent's private key.
  - status is a read-only AgentBook lookup -> { registered, humanId, contract, network }
  - API_URL env var overrides the relay base URL.

THE SCALING PROBLEM (key realization) — "what if we need 100 agents?"
  Doing register 100x is the WRONG model, for two reasons:
    1. 100 interactive World App approvals (the human proof is per-registration).
    2. More importantly: every address YOU register resolves to the SAME humanId,
       because there is one of you. 100 registrations = 1 human backing 100 wallets,
       NOT 100 verified humans. The gate counts usage per (endpoint, humanId), so the
       human-ness is pooled, not per-wallet.

THE MODEL WE'RE ADOPTING INSTEAD
  Matches plan §3: "verification attaches to a lineage, not each ant."
  - Register ROOTS, not ants. One registration per REAL human. Recruit N humans -> N
    verified lineage roots (N distinct humanIds). That's the whole verified cohort.
  - Children inherit "verified" through OUR bookkeeping (colony state + ENS records),
    not through new World App proofs.
  - Anonymous lineages register nothing. That's the verified-vs-anon experiment (plan §8).
  - Gateway pattern (to confirm): route gated/x402 requests through ONE registered
    gateway wallet per lineage; individual ants are internal identities the gateway acts
    on behalf of. Human-backing + payments live at the gateway. Collapses "100 taps" ->
    "1 tap per human" = tens of registrations for tens of humans (plan: "tens, not thousands").

EXPLORATION RESULTS (2026-06-13) — server-side gate mechanics
  Sources: github.com/worldcoin/agentkit, the world-x402-agentkit-example repo, and
  World's AgentKit announcement coverage. (Full server internals live in the repo's
  x402/DOCS.md and skills/agentkit-x402/SKILL.md — not yet deep-read.)

  CONFIRMED — what the gate checks:
    Request flow: hit endpoint -> 402 challenge -> agent signs a CAIP-122 message
    (eip191) with its wallet -> server verifies the signature and looks up THAT signing
    wallet in AgentBook -> resolves to a humanId -> applies policy. The ONLY thing the
    server keys on is the signing wallet address.

  CONFIRMED — the hard invariant:
    One human = one humanId, across ALL their wallets (a one-way hash tying every wallet
    you register back to the single you). You can NEVER produce more than one humanId
    from one human. No Sybil cheat exists, by design. So registering 100 wallets to
    yourself = 100 wallets, still 1 humanId.

  ANSWER to "what if we need 100 agents?":
    - Register-per-ant: 100 World App taps AND they all collapse to your 1 humanId.
      Strictly worse, no reason to do it.
    - One registered wallet signs for the whole lineage: server only checks the SIGNING
      wallet, so ants can sign with their lineage's single registered wallet. One tap per
      human, not per ant. >>> INFERENCE to validate by test (see below); the docs describe
      "each agent registers its own wallet" and do NOT officially bless one-wallet-many-agents.
    - AgentKit's built-in free-trial limit (3 uses / endpoint / humanId) is POOLED per
      human — but mostly irrelevant because Colony runs its OWN x402 ClickHouse gate and
      sets its own policy.

  RECOMMENDED COLONY ARCHITECTURE (decided):
    World ID verification happens ONCE per real human -> yields a humanId -> that humanId
    IS the "verified lineage" badge, stamped into ENS / colony state. Ants inherit
    verified-ness through OUR bookkeeping, never by replaying World ID. When an ant must
    prove human-backing to a gated endpoint, it signs with its lineage's registered wallet.
    Matches plan §3 ("verification attaches to a lineage, not each ant"), now confirmed
    against real mechanics.
