BUILD-FIX HISTORY (the two failures we hit + fixes)
  1. First deploy built the WRONG Dockerfile: log showed `npm --prefix arc ci` +
     nodejs — Railway was auto-picking the repo-ROOT Dockerfile, not ours.
     Fix: set the service to clickhouse_api (Root Directory).
  2. Then clickhouse_api/Dockerfile failed: `COPY clickhouse ./clickhouse ->
     "/clickhouse": not found`. Cause: the old Dockerfile used REPO-ROOT-relative
     COPY paths, but Root Directory=clickhouse_api makes the build context that
     folder. Fix: rewrote the Dockerfile self-contained (COPY requirements.txt . ;
     COPY . . ; uvicorn main:app — no --app-dir, no COPY clickhouse) + added
     clickhouse_api/.dockerignore (.venv,*.log,.env) + clickhouse_api/railway.toml.
  3. Success: container "Application startup complete", uvicorn on 0.0.0.0:8080,
     healthcheck GET /health 200.

GOTCHAS LEARNED (Railway, for the team)
  - uvicorn logs to STDERR -> Railway tags them [err] even though they're INFO/success.
    "Application startup complete" under [err] is NORMAL, not an error.
  - PORT is injected by Railway (saw :8080); Dockerfile CMD must use ${PORT:-8009}.
  - /health returns 200 EVEN IF ClickHouse is unreachable (body {ok:false}); a green
    healthcheck only proves the app is up — check the BODY for clickhouse:"reachable".
  - The healthcheck client IP 100.64.0.2 is Railway's INTERNAL prober (CGNAT range
    100.64.0.0/10), not our API's IP. Public access is via the *.up.railway.app domain.
  - .dockerignore is relative to the build context; with Root Directory=clickhouse_api,
    the repo-root .dockerignore does NOT apply -> we added clickhouse_api/.dockerignore.
