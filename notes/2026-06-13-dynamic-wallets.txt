KEY ARCHITECTURE FINDING — this env uses V3 (MPC / WaaS) wallets
  - The deprecated server endpoint /environments/{env}/embeddedWallets creates V2/Turnkey
    wallets and is REJECTED here: "Cannot create a Turnkey embedded wallet in an environment
    configured for V3 wallets."
  - The working server-side endpoint for V3 pregenerated wallets:
      POST {base}/environments/{ENV_ID}/waas/create
      headers: Authorization: Bearer <DYNAMIC_API_KEY>, Content-Type: application/json
      body:    { "identifier": "<email>", "type": "email", "chains": ["EVM"] }
      -> 201, returns user.verifiedCredentials[].address (eip155 EVM address),
         wallet_properties.version "V3", thresholdSignatureScheme "TWO_OF_TWO".
  - V3 = MPC: key shares held by Dynamic (2-of-2). **No raw private key is returned.**
    Signing happens THROUGH Dynamic (API/SDK), not via a local private key. This is the big
    contrast with our cast-generated treasury wallet (which we DO hold the key for).

WHAT THIS MEANS FOR COLONY WALLET ARCHITECTURE
  - Dynamic can pregenerate many ant wallets fast, server-side, with one API key. Good for
    "tens of ants" and beyond. Each wallet is tied to an identifier that a real user could
    later "claim" by logging in with that identifier.
  - Tradeoff vs. local cast keys:
      * Dynamic MPC: managed, no key to leak, claimable by humans, but SIGNING requires a
        Dynamic API/SDK call (not a local signer). Need to wire that for ant Arc txs.
      * cast local keys (treasury-style): we hold the key, can sign locally/instantly, but we
        manage secrets ourselves and there's no human-claim story.
  - Open question: how to SIGN an Arc tx from a Dynamic V3 wallet server-side (the WaaS signing
    API). Needed before ants funded via Dynamic can actually stake/transfer on Arc.
